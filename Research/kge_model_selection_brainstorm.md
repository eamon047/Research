# KGE Model Selection Brainstorm

## 背景

当前讨论的问题是：不同知识图谱嵌入模型是否在不同结构模式下具有稳定的优势，如果有，能否先识别 query 或 relation 的结构特征，再为其选择更合适的模型，从而提升整体链路预测效果。

初步结论是，这个方向是合理的，但更适合从图结构和关系模式切入，而不是从实体名字或答案词面内容切入。原因在于标准 KGE 任务中，实体通常以 ID 表示，模型学习的重点不是名称语义，而是图中的结构规律。

因此，一个更稳妥的研究问题可以表述为：

> 不同 KGE 模型在不同关系结构模式下是否存在稳定的性能差异？如果存在，能否基于这些结构特征进行模型选择？


## 对选题的判断

这个想法可以成立，但最好按两步走：

1. 先证明模型之间确实存在可利用的互补性。
2. 再尝试设计模型选择器。

不建议一开始就直接做“分类器选模型”，因为如果模型之间的差异不够稳定，分类器很容易变成一个没有实际增益的工程堆叠。

更稳的论文路径是：

- 先做分析型研究：比较不同模型在不同关系模式下的表现差异。
- 如果差异显著，再扩展成方法型研究：基于结构特征做模型路由或模型选择。


## 为什么不建议从实体名称或答案内容切入

例如“城市”“人名”“国家”这类标签，在 KGQA 中可能有帮助，但在纯 KGE 实验中通常不是核心变量。原因包括：

- KGE 模型训练和推理时主要接收实体和关系 ID。
- 数据集中实体名经常被映射为无语义的符号，例如 `/m/0p5mw`。
- 在这种设定下，词面内容本身并不能稳定解释模型差异。

如果未来引入文本描述、实体类型信息或 LLM 语义特征，那么“答案内容类别”可以重新成为研究变量；但那已经属于 KGE 与文本语义结合的问题，不是纯 KGE 的主线。


## 最值得尝试的分类维度

下面这些维度都满足两个条件：

- 可以从图结构中量化出来
- 有理由影响不同 KGE 模型的表达能力或优化难度

### 1. Relation Mapping Type

典型类别：

- 1-1
- 1-N
- N-1
- N-N

含义：

- `1-1`：一个 head 通常只对应一个 tail，反过来也类似。
- `1-N`：一个 head 往往对应多个 tail。
- `N-1`：多个 head 对应同一个 tail。
- `N-N`：两边都存在多重映射。

量化方式：

- 对每个关系 `r`，统计：
  - 平均每个 head 对应多少个 tail
  - 平均每个 tail 对应多少个 head
- 根据阈值将关系归到四类。

为什么重要：

- 这是 KGE 里最经典、最稳的结构分类。
- 不同模型在多值映射上的表达能力不同。
- 例如 `TransE` 往往更容易在 `1-N / N-1 / N-N` 关系上受到几何约束。

建议：

- 这是最应优先做的主分析维度之一。


### 2. Symmetry / Antisymmetry

含义：

- 对称关系：若 `(h, r, t)` 成立，则 `(t, r, h)` 也经常成立。
- 反对称关系：若 `(h, r, t)` 成立，则 `(t, r, h)` 通常不成立。

量化方式：

- 对每个关系统计对称闭包比例：
  - 在所有 `(h, r, t)` 中，有多少满足 `(t, r, h)` 也存在。

为什么重要：

- 某些模型天然更适合对称关系，某些模型更能表达反对称关系。
- 例如 `DistMult` 通常更偏向对称模式。
- `ComplEx`、`RotatE` 通常对反对称关系更有表达力。

建议：

- 非常适合作为和 mapping type 并列的核心分析维度。


### 3. Inverse Relation Existence

含义：

- 是否存在关系对 `(r1, r2)`，使得 `(h, r1, t)` 往往对应 `(t, r2, h)`。

量化方式：

- 对关系对统计逆匹配比例。
- 为每个关系定义 inverse strength，表示其逆关系证据强弱。

为什么重要：

- 模型对逆关系的建模能力存在明显差异。
- `RotatE` 通常被认为在这类模式上更有优势。

建议：

- 这是非常值得做的第三个主特征。


### 4. Relation-Specific Answer Cardinality

含义：

- 在特定关系下，给定 `(h, r, ?)` 时通常有多少正确答案。
- 或给定 `(?, r, t)` 时通常有多少正确答案。

量化方式：

- 对每个关系统计：
  - `avg_tail_per_(h,r)`
  - `avg_head_per_(r,t)`

和 mapping type 的区别：

- mapping type 是离散分类。
- answer cardinality 是连续值，更细粒度。

为什么重要：

- 它直接刻画 query 的多解性。
- 多解关系和单解关系上，不同模型的 ranking 行为可能不一样。

建议：

- 很适合做分类器输入特征，也适合做连续统计分析。


## 很有用的辅助维度

### 5. Relation Frequency

含义：

- 某关系在训练集中出现的次数。

为什么重要：

- 高频关系通常训练更充分。
- 低频关系更依赖模型归纳偏置。
- 某些模型在稀疏数据上更稳，某些模型需要更多样本。

建议：

- 更适合作为辅助特征或控制变量，而不是论文的唯一主轴。


### 6. Head/Tail Entity Degree

含义：

- 实体连接多少条边，可以统计全局 degree，也可以统计关系条件下的 degree。

为什么重要：

- 高 degree 实体通常更像 hub，容易带来更高的歧义性和负采样难度。
- 不同模型对 hub 实体的鲁棒性可能不同。

建议：

- 适合做 query-level 特征。
- 作为 relation-level taxonomy 的主轴时解释力略弱。


### 7. Query Difficulty

含义：

- 给每个测试 query 定义一个结构难度，而不是笼统地说“这个问题难”。

可选量化方式：

- 正确答案数量
- head 或 tail 的 degree
- 关系下尾实体分布熵
- 局部邻域复杂度
- 相似候选实体的干扰强度

为什么重要：

- 如果最终目标是为单个 query 选模型，那么 query difficulty 是一个很自然的中间变量。

建议：

- 值得做，但前期要尽量采用简单、可解释的定义。


## 更有研究味的扩展维度

### 8. Composition Pattern

含义：

- 是否存在 `r1 ∘ r2 ≈ r3` 这样的关系组合模式。

为什么重要：

- 不同模型对组合关系的隐式表达能力可能不同。

难点：

- 计算较重，噪声也会更大。

建议：

- 适合作为增强分析，不建议最先做。


### 9. Relation Entropy

含义：

- 某关系的尾实体分布或头实体分布是集中还是分散。

为什么重要：

- 低熵关系可能更容易被简单偏置学到。
- 高熵关系更考验真正的结构建模能力。

建议：

- 适合作为 answer cardinality 的补充连续特征。


### 10. Type Constraint Strength

含义：

- 某关系对 head/tail 的类型约束是否很强。

例如：

- `born_in` 的 head 往往是 Person，tail 往往是 Location。

为什么重要：

- 类型约束强的关系通常更容易预测。
- 某些模型可能更依赖这种隐式类型边界。

建议：

- 如果数据集中存在类型信息，值得尝试。
- 如果没有类型信息，也可用观测到的头尾实体集合集中程度近似。


### 11. Relation Pattern Purity

含义：

- 某个关系是否主要表现为一种清晰的结构模式，还是混合了多种模式。

为什么重要：

- “干净”的关系模式更容易被特定几何结构捕捉。
- 混合模式关系通常更难学。

建议：

- 是一个很适合论文讨论的派生指标。


### 12. Path Redundancy / Local Structural Support

含义：

- 一个三元组是否可由多条路径支持。
- 某 query 周围是否存在丰富的局部图证据。

为什么重要：

- 冗余结构高的事实可能更容易被恢复。
- 不同模型利用局部路径证据的能力可能不同。

建议：

- 更适合做 query-level 路由特征。


## Relation-Level 与 Query-Level 的两种建模思路

### Relation-Level 分类

输入：

- relation 的统计特征，例如 mapping type、symmetry、inverse strength、frequency

输出：

- 哪个模型更适合这个 relation

优点：

- 简单
- 容易解释
- 适合毕业论文第一版

缺点：

- 太粗粒度，同一关系内部不同 query 的差异可能被忽略


### Query-Level 分类

输入：

- 单个 query 的局部结构特征，例如 degree、answer cardinality、局部路径支持、query difficulty

输出：

- 哪个模型更适合这个 query

优点：

- 更细粒度
- 潜在提升空间更大

缺点：

- 噪声更大
- 设计和验证都更复杂


## 当前最推荐的起步方案

如果要尽快形成一套稳妥的毕业论文主线，建议优先只做下面 4 个核心维度：

1. relation mapping type
2. symmetry / antisymmetry
3. inverse relation strength
4. relation-specific answer cardinality

原因：

- 这些维度最容易量化
- 最容易和 KGE 模型表达能力联系起来
- 最有希望真正拉开 `TransE / RotatE / ComplEx / HElix` 等模型的差异
- 也最容易写进论文方法和实验分析部分


## 一个比较稳的论文推进路径

### 第一阶段：验证互补性是否存在

- 在不同关系模式下比较各模型的 MRR / Hits@k
- 证明模型之间确实存在稳定差异

### 第二阶段：定义可预测的结构特征

- 从 relation-level 开始
- 先做简单、可解释的特征

### 第三阶段：尝试模型选择器

- 先做 relation-level selector
- 如果效果明显，再尝试 query-level selector


## 一句话总结

这个研究方向是成立的，但最适合从“关系结构模式差异”切入，而不是从“实体名称或答案词面内容差异”切入。论文上建议先做分析型研究，证明不同 KGE 模型对不同结构模式确实存在稳定偏好；如果互补性明显，再进一步设计基于结构特征的模型选择器。
